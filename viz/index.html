What you're seeing

This simulator is the same architecture as the C++ project. Tasks are submitted to a priority queue protected by a mutex + condition variable. Each Worker is one std::thread that sleeps on the cv until a task arrives, pops the highest-priority one, calls execute(), and loops.

Try this: start a long ComputeTask running on every worker, then submit a pri 10 PrintTask. It can't preempt a running task — but it jumps to the front of the queue, so it runs the moment any worker finishes. That's exactly how the real std::priority_queue-backed thread pool behaves.

Encapsulation hide data

What: Bundle data with the methods that operate on it. Outside code can't reach in and corrupt internal state.

In ThreadForge: Task::status_ is private and atomic. Outside callers can read via getStatus(); only the task itself can transition via the protected markRunning/Completed/Failed() helpers.

class Task {
private:
    std::atomic<TaskStatus> status_;       // hidden
protected:
    void markRunning()   { status_.store(TaskStatus::Running); }
public:
    TaskStatus getStatus() const { return status_.load(); }
};

Interview line: "The class owns its invariants — outside code can observe, but can't corrupt."

Abstraction hide complexity

What: Expose what the class does, hide how. Callers code against an interface, not an implementation.

In ThreadForge: Task::execute() is pure virtual. The ThreadPool calls task->execute() with no idea whether it's a print, a compute, or a failing task.

Distinction from encapsulation: Encapsulation hides data. Abstraction hides complexity.

Inheritance is-a

What: A derived class acquires the base's interface and behavior, then extends or overrides them.

class PrintTask : public Task {       // PrintTask IS-A Task
public:
    PrintTask(std::string m) : Task("PrintTask"), msg_(std::move(m)) {}
    void execute() override {              // override catches typos
        markRunning();
        std::cout << msg_ << "\n";
        markCompleted();
    }
};

Watch out: Prefer composition over inheritance when the relationship is "has-a". Deep inheritance hierarchies become fragile.

Polymorphism vtable

What: One interface, many forms. A base pointer or reference can point to any derived type, and the right method runs at runtime via the vtable.

std::vector<std::shared_ptr<Task>> tasks;
tasks.push_back(std::make_shared<PrintTask>("hi"));
tasks.push_back(std::make_shared<ComputeTask>(42));

for (auto& t : tasks) {
    t->execute();   // correct execute() per type — runtime dispatch
}

vtable: A per-class table of function pointers. When you call a virtual function, the CPU looks up the pointer in the actual object's vtable. Cost: one extra indirection per virtual call.

SOLID, mapped to this project

S — Single Responsibility. Task = work definition. Queue = thread-safe storage. Worker = consume loop. Pool = lifecycle. Each class has one reason to change.

O — Open/Closed. Add a NetworkTask, DatabaseTask — no changes to Task.h, Worker.cpp, or ThreadPool.cpp.

L — Liskov Substitution. Any subclass passed as Task* must call markRunning() then exactly one of markCompleted/Failed(). Skipping these violates LSP.

I — Interface Segregation. The base interface is just execute() + describe(). We don't force every task to implement pause(), cancel(), etc.

D — Dependency Inversion. ThreadPool depends on the Task abstraction, not on any concrete subclass.

Mutexes & RAII

A mutex ensures only one thread enters a critical section. RAII ties lock lifetime to scope: lock_guard locks in its constructor and unlocks in its destructor — even if your code throws.

{
    std::lock_guard<std::mutex> lock(mtx);   // locks here
    counter++;                               // critical section
}                                            // auto-unlocks here

lock_guard vs unique_lock: lock_guard is simple, can't be unlocked manually. unique_lock is needed for condition variables (it can hand off the lock during cv.wait).

Condition Variables

A cv lets a thread sleep until another signals state has changed — instead of spinning the CPU. Always use a predicate; threads can wake spuriously.

// Consumer:
std::unique_lock<std::mutex> lk(mtx);
cv.wait(lk, []{ return dataReady; });   // re-checks predicate
process();

// Producer:
{
    std::lock_guard<std::mutex> lk(mtx);
    dataReady = true;
}
cv.notify_one();

This is exactly what ThreadSafeTaskQueue::wait_and_pop does — sleep until queue non-empty or shutdown.

Deadlock circular wait

Two threads, two locks, acquired in different orders → both wait forever.

Prevent it: always acquire locks in the same order, or use std::scoped_lock(m1, m2) (C++17) which acquires multiple atomically.

std::atomic

For single variables (counters, flags, status), atomic is faster than a mutex and lock-free for simple types.

std::atomic<int> counter{0};
counter++;            // atomic fetch-and-add
counter.load();       // atomic read

atomic vs mutex: atomic for one variable; mutex for blocks of code or multiple variables that must change together.

Producer-Consumer Pattern

The pattern at the heart of ThreadForge — and most real systems (web servers, kernels, message brokers).

Producers push tasks. The queue's cv.notify wakes one waiter. The waiter pops and executes. Shutdown sets a flag and notify_all so waiters can exit.

C++ specifics they'll probe

      
    
        
  • Pointer vs reference: a reference must bind on creation and can't rebind; a pointer can be null and reassigned. Use references when "must point to something"; pointers when "may not, or may change".
    • 
        
  • unique_ptr vs shared_ptr: unique = sole ownership, move-only, zero overhead. shared = reference-counted, atomic refcount, bigger and slightly slower.
    • 
        
  • Rule of 3 / 5 / 0: if you define one of (dtor, copy ctor, copy assign), define all 3. With move ops, 5. Better: design your class so RAII members handle everything → rule of 0.
    • 
        
  • Virtual destructor: required on any base class deleted through a base pointer. Without it, derived destructors don't run → leaks.
    • 
        
  • Move semantics: std::move doesn't move — it casts to rvalue so a move ctor / move assignment can take ownership of resources.
    • 
        
  • const correctness: a const member function promises not to modify the object. Declare every getter const.
    • 
      

    


    

      
Linux fundamentals

      
    
        
  • Process vs thread: process owns the address space; threads share it within a process.
    • 
        
  • fork vs exec: fork duplicates the process, exec replaces the program image. Shells do fork + exec for every command.
    • 
        
  • Signals: async notifications. SIGTERM = polite stop, SIGKILL = uncatchable, SIGSEGV = invalid memory access.
    • 
        
  • gdb: gdb ./binary, run, on crash bt for backtrace, frame N, print var.
    • 
        
  • valgrind: valgrind --leak-check=full ./binary finds leaks and use-after-free.
    • 
        
  • strace / ltrace: trace syscalls / library calls of a running process.
    • 
        
  • top / htop / perf: CPU and memory at a glance; perf for hardware counters.

Real-time systems concepts

      
    
        
  • Hard real-time: missing a deadline = system failure. Hard RT systems use deterministic scheduling and avoid heap allocation in the hot path.
    • 
        
  • Soft real-time: missing a deadline degrades quality but isn't catastrophic (video playback).
    • 
        
  • Jitter: variation in timing of a periodic event. Hard RT minimizes jitter.
    • 
        
  • Priority inversion: a low-priority thread holds a lock a high-priority thread needs. Solved by priority inheritance protocols.
    • 
        
  • Lock-free: guaranteed at least one thread makes progress (CAS-based queues, atomics).
    • 
        
  • Bounded execution: every code path has a known max runtime — no unbounded loops, no waiting on the network in the deadline path.
    • 
      

    


    

      
Debugging a segfault (from scratch)

      
    
        
  1. Compile with -g -O0 -fsanitize=address,undefined.
    2. 
        
  2. Run under gdb: gdb ./bin then run.
    3. 
        
  3. On crash: bt (backtrace), frame N to inspect a level, print var to see values.
    4. 
        
  4. Check the obvious: dereferencing a null/dangling pointer, out-of-bounds array access, double-free, use-after-free.
    5. 
        
  5. If multithreaded: rerun with -fsanitize=thread to surface data races.
    6. 
        
  6. If sporadic: log + reproduce minimally, then attach gdb again with a watchpoint on the corrupted variable.
